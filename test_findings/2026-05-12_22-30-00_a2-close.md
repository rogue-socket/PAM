## Three commits, separable revert

### Commit 1 — `91b1b98` — assembly score-sort, narrow scope

`_support_path_result_nodes` now sorts the full candidate set by `node_scores` with a small `RELATIONSHIP_PRIORITY_BONUS` (0.1, env-overridable) for support-path endpoints. `_relationship_result_nodes` left unchanged — its edge-iteration contract (source-first for incoming, target-first for outgoing) is the right shape for explicit-relation queries like SUPERSEDES, and modifying it broke `test_ranker_relationship_query_keeps_best_edge_endpoints_in_results` in a way that revealed the contracts are legitimately different.

Effect: idx 86, 87 → rank 1 (gold's score 0.717/0.759 dominates the 0.318–0.494 support-path endpoints). idx 81 unchanged (no score to sort by).

### Commit 2 — `a8a5592` — DERIVED_FROM score propagation

For each DERIVED_FROM edge in `combined_inter_edges` where seed text contribution ≥ 0.15 and target text contribution ≤ 0.05, add `α=0.5 * seed_text` to `node_scores[target]`. Asymmetric gate is the surface-area control — when both endpoints carry the FTS hit (Hard's literal-anchor lookups), the trigger never fires. Tunables: `PAM_DERIVED_PROP_ALPHA`, `PAM_DERIVED_PROP_SEED_FLOOR`, `PAM_DERIVED_PROP_SINK_CEILING`.

Effect on idx 81: seed text = 0.278 → boost = 0.139 → gold rank 11 → 8 in `ordered_nodes` (score 0.273 → 0.412).

### Commit 3 — `243c3ea` — drop the directional hard filter

Discovered while debugging idx 81's Claude eval miss: gold was rank 8 in `ordered_nodes` but Claude still returned NO_ANSWER. The eval prompt renderer (`scripts/run_copilot_cli_eval.py:_render_retrieval_context`) only surfaces `result.relationships`, not `result.graph_explanations`. The seed→gold DERIVED_FROM edge was in `graph_explanations` but not in `relationships` because `_rank_relationship_hits` was using `directional_edges or filtered_edges` — when any directional match existed in the pool, non-directional edges were dropped. The seed→gold edge fails direction match (gold is graph-expanded, not in `candidate_ids`), so it lost the slot.

The fix: just sort `filtered_edges` directly. Direction is already a +0.35 sort bonus in `_score_relationship` — the hard filter was redundant on the rank side and lossy on the surfacing side.

Effect: idx 81 Claude eval miss → PASS.

## Why the morning's framing missed this

The 2026-05-12 wrap concluded "PAM is already graph-prioritized for relationship mode — the failure is that the existing edges aren't typed/weighted strongly enough" and parked the structural fix behind Phase B. That diagnosis was half right: the assembly layer IS graph-prioritized, but it prioritizes by edge-iteration order, not by score-of-connected-node OR by edge type. The ablation branches were operating in the rank-key layer (which the assembly bypasses), so all three returned 0–1/3 targets despite correct internal math.

The shipped fix doesn't add typed edges. It uses the existing DERIVED_FROM edge for what it's for: a signal that the target's relevance is the graph link itself, not the keyword overlap.
