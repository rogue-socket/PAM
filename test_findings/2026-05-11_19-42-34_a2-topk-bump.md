# Phase A.2 — TOP_K bump replaces the floor sweep (2026-05-11)

Phase A.2 was scoped on 2026-05-09 as a coordinate-descent weight sweep over `VEC_SIMILARITY_FLOOR`, `w_vec`/`w_text`, and candidate-pool sizes. The actual fix is **one knob: `TOP_K = 10 → 15`**. The floor and weight sweep were solving the wrong problem.

### 2. Cell 1 — VEC_SIMILARITY_FLOOR=0.60

**Detailed 99/110 raw (~101/110 after matcher-FN triage), IRL 52/68 (colloquial 6/16).**

Floor=0.60 culled vector candidates in the 0.5–0.6 cosine range. This:

- Removed the noisy "adaptive orchestration / launch-target" candidates from the top-10 on the failing detailed relationship queries, letting the gold note (e.g., community workshop transcript) win → idx 81, 86 flipped to PASS.
- Also culled the cosine-0.5–0.6 vector candidates that the IRL colloquial_relationship row depends on (`"who's my manager"` → 1:1 with Anya note at cosine ~0.55) → colloquial 13/16 → 6/16. **Catastrophic.**

The two gates moved inversely with floor. **Rejected.**

The Cell-0 failure mode was **not** pool composition or weight tuning — it was **rank-by-recency burying older textually-relevant gold notes**. The gold notes already exist in the candidate pool; they just lose rank slots to high-recency vector pollution. Floor=0.60 worked by accident (it removed the polluters from the pool), but at the cost of removing useful colloquial-recall vector hits at the same cosine range.

## Ship attempt — REVOKED after Hard regression

Initially shipped as `TOP_K=15` (commit pending). Hard / Large spot-check the same session caught a catastrophic regression on Hard:

| Suite | TOP_K=10 (Cell 0) | TOP_K=15 | Δ |
|---|---|---|---|
| Detailed | 93/110 (~94 real) | 98/110 (~100 real) | +5 raw |
| IRL | 59/68 | 62/68 | +3 |
| Hard | **192/192** | **148/192** | **−44** |
| Large | not run | (not run) | — |

Hard returned NO_ANSWER on 44 queries across all five categories (lookup, paraphrase, relationship, timeline, negative). Sample regressions: idx 7 `"What source was derived from the revised Harbor Ledger plan?"`, idx 19 `"What source was derived from the revised Harbor Lattice plan?"`, idx 21 timeline, idx 29 paraphrase. All 100% at TOP_K=10.

**Reason:** Hard is constructed with tight literal-anchor lookups against a bigger corpus (~192 q vs detailed's ~110, larger note pool). The 5 extra context entries at TOP_K=15 introduce candidates whose literal anchors don't match the query, and Claude's eval prompt rule ("If retrieval result does not support an answer, reply exactly with NO_ANSWER") trips into refusal more often when context mass grows past a threshold. Detailed and IRL didn't show this because:
- Detailed queries actively need the extra rank slots (buried gold at rank 11+).
- IRL colloquial winners are already top-1 — extra context is benign.
- Hard queries had everything they needed at top-10; extra context is pure noise.

**Net across all measured suites at TOP_K=15: −36 hits.** A clear lose.

`config.py:9` reverted to `TOP_K = int(os.getenv("PAM_TOP_K", "10"))`. The `PAM_TOP_K` env override stays in place for future per-call tuning.

## What this means for A.2

Phase A.2 is **not resolved.** TOP_K is a single global knob and there is no single value that wins all suites:

- TOP_K=10: Hard 192/192, IRL 59/68, detailed 93/110 (−3 below 96 floor).
- TOP_K=15: Hard 148/192 (−44), IRL 62/68 (+3), detailed 98/110 (+5).

The diagnostic insight (gold notes for detailed misses live at rank 11+ at TOP_K=10, pushed there by high-recency vector pollution) is still valid. The lever that addresses it without the cross-suite damage is **probably rank-weight, not pool size**. Specifically:

- **Lower `WEIGHT_RECENCY` from 0.30 → 0.20.** The diagnosed pollution is recency-driven. Reducing recency weight should lift the older-but-relevant gold notes within top-10 directly. Should be neutral on Hard (Hard's gold notes are recent within their own corpus, so their relative recency-rank doesn't change much — but verify).
- **Add `PAM_W_RECENCY` env override** (parallel to the other Phase A.2 env overrides) so this is one env-prefixed invocation per cell.

Next attempt: one cell at `PAM_W_RECENCY=0.20`, all four suites (detailed + IRL + Hard + Large). If Hard holds and detailed lifts, ship. If Hard regresses, this round of A.2 ends without a fix and we park behind Phase B.
