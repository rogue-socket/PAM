## Misses, grouped by root cause

### A. Wrong-premise correction is missing — answer-LLM punts (4 queries, ~5.9 points)

| idx | query | expected | got | retrieved? |
|---|---|---|---|---|
| 18 | what did Diego decide about the auth refactor? | "did not decide" / "shadowed" / "mentoring" | NO_ANSWER | ✅ yes (diego_shadow + mentor notes top) |
| 19 | the bug Stripe caused in production | "sandbox" / "not production" / "throttling" | NO_ANSWER | ✅ yes (stripe_flake_v2 + stripe_mock) |
| 20 | the database migration we shipped last week — how's it going? | (expected to push back) | NO_ANSWER | ✅ yes (event-store decision/RFC top) |
| 21 | the cache fix Anya wrote — what was the approach? | "Rakhi" / "per-key" | NO_ANSWER | ✅ yes (rakhi_chat + pr_443_review) |

PAM's retriever is doing its job — every relevant note is in the top‑10. The grading LLM refuses to commit because the question's premise (Diego decided, Stripe in prod, migration shipped, Anya wrote the cache fix) doesn't textually match anything. NO_ANSWER is being used as "I'm not sure" rather than "the premise is wrong; here's what actually happened."

**Fix-level**: this is an *answer-generation prompt* problem, not a retrieval problem. The chat-agent / grading prompt needs an explicit rule: *when retrieved context contradicts the premise, state the contradiction*. Cheapest unlock for ~6 points.

### B. Multi-hop entity bridge (1 query, idx 15)

> "what was the original timeline for the project Mira pushed back on?" → expects "3 weeks"

Retrieved top-10 includes `rfc_mira_pushback` (Mira → ClickHouse RFC) **but not** `rfc_clickhouse` (the RFC text containing "Estimated 3 weeks"). The `Updated event-store migration estimate after Postgres+ClickHouse split: 6 weeks, not 3` note is there — supersedes the original — but the original (`rfc_clickhouse`) isn't surfaced.

**Why**: FTS on "original / timeline" hits the supersession note (which uses "6 weeks, not 3"); the original 3-week RFC uses the word "Estimated", not "original" or "timeline". The bridge is Mira → pushback note → ClickHouse entity → RFC. `graph_expander` should walk REFERS_TO from the pushback note to the RFC via the ClickHouse entity. The fact that this miss survives means either (a) the ClickHouse entity isn't being extracted at ingest, or (b) the expander isn't traversing past one hop on this shape.

**Fix-level**: verify entity extraction wrote a ClickHouse entity node for both `rfc_clickhouse` and `rfc_mira_pushback`; if so, raise the expansion budget so 2-hop entity bridges land in top-10. This is the *intended* graph-native behavior — and the only multi-hop query the corpus has that breaks; if it's an extraction gap, it's a one-line config fix.

### C. Colloquial role/relationship — no aggregation primitive (3 queries)

| idx | query | expected | got |
|---|---|---|---|
| 32 | who's my most frequent collaborator? | Rakhi | Diego (single high-recency mention dominates) |
| 33 | who critiques my code? | Anya | NO_ANSWER (auth_pr_event with "requesting Anya for review" was NOT retrieved) |
| 56 | who do I pair with on code? | Rakhi | Diego ("Plan: pair on a real bug first" lexically matches "pair") |

These need different fixes:
- **idx 32 ("most frequent")** is a frequency-aggregation question. PAM has no "count co-occurrences across the workspace" operator. The retriever surfaces what's most recent/most relevant *to one hit*, not "who appears the most across many hits."
- **idx 33 ("who critiques my code")** is a retrieval gap — the one note that answers it (`auth_pr_event`: "requesting Anya for review") wasn't in the top-10. FTS terms `critiques` and `review` don't connect; vector recall didn't bridge it.
- **idx 56 ("who do I pair with")** is a lexical foot-gun. The `diego_mentor_note` literally contains "Plan: pair on a real bug first" — so any keyword match on "pair" pulls Diego over Rakhi. The semantic truth (Rakhi was an actual coding collaborator per `rakhi_chat`: "We sketched per-key locks") doesn't dominate ranking.

**Fix-level**: (a) a relationship-aggregation pass for role-shaped questions ("who … most"), (b) better recall on synonym pairs (critique ↔ review), (c) a "past vs plan" signal so future-tense ("plan to") deranks against past-tense action.

### D. Time-relative against real-world today (2 queries)

| idx | query | expected | got |
|---|---|---|---|
| 25 | what did I do last week? | "Diego" / "shadow" | NO_ANSWER |
| 27 | what was I working on two weeks ago? | "Anya" | Answer about event store migration — semantically reasonable, doesn't contain "Anya" |

Today is 2026-05-19; corpus runs 2026-03-28 → 2026-04-30. "Last week" = 2026-05-12 → 2026-05-19, which has zero corpus items. The query parser is binding the window literally; retrieval returns the usual top items but the grader sees nothing matching "last week" and either punts (idx 25) or describes the most-recent items without nailing the specific Anya/Diego mention (idx 27).

**Fix-level**: when a time-window filter produces no results, the parser should either fall back to "most-recent N items" with a note ("no items in last week — showing most recent") or be calibrated against the corpus's effective "now". The fixture has no notion of "as-of-time" for the user, which is a corpus design quirk; in production this same query against fresh memory wouldn't fail.

### E. Edge direction confusion (1 query, idx 57)

> "who reports to me?" → expected NO_ANSWER (user has no direct reports); got "Anya"

The single signal is `manager_1on1` ("1:1 with Anya. … She said the staff promo packet is on track."). LLM mis-inferred direction: Anya gives feedback to user ⇒ Anya is the manager, not a report. Mentor relationship with Diego is correctly modeled as one-way (mentor, not report).

**Fix-level**: this is the colloquial-relationship axis from the fixture's design — testing whether retrievers confuse edge direction. PAM has no direction-typed edge for management / reporting. A `MANAGES` or `MANAGED_BY` typed edge written from 1:1 phrasing would fix this category structurally. Otherwise, a prompt nudge ("when asked about reports / reportees, only answer if explicit reporting language was used").
