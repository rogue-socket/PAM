## Open findings

Verified against the code. Ordered by impact on kayo.

### O1 — Telemetry log writes are non-transactional with DB commits
**Where:** `pam/feedback.py:23-26`, `pam/lifecycle.py:17-21`, `pam/ingestion/pipeline.py:48-51` (and similar JSONL append sites).
**Issue:** Each event is written by opening `LOG_PATH` in append mode, writing one JSON line, and closing — no `flush()` + `fsync()`, no atomic temp-file-rename, and no participation in the SQLite transaction. A crash between the DB commit and the log write (or mid-line) leaves PAM_LOG.jsonl out of sync with `pam.db` and may produce a torn JSON line that breaks downstream readers.
**Why it matters for kayo:** kayo runs unattended; a single crash silently corrupts the audit trail kayo depends on for self-introspection.
**Fix sketch:** add a `_append_log_line()` helper that flushes + `os.fsync()`, and document log loss as best-effort below DB commits.

### O2 — `check_database_health()` is never called automatically
**Where:** `pam/db/schema.py` (defined) — no callers in `pam/`.
**Issue:** FTS drift (missing or orphan rows) is detectable but not surfaced. `get_initialized_connection()` does not invoke it. After enough ingest+feedback churn, FTS results will silently degrade.
**Fix sketch:** invoke health check on first connection per process (lazy, behind a flag to avoid hot-path cost) and log a warning; optional auto-rebuild of `fts_index` when drift count exceeds a threshold.

### O3 — Two write paths for `SUPERSEDES` / `CONTRADICTS` with divergent side effects
**Where:** explicit feedback path `pam/feedback.py:110-147` (the `supersede()` command) vs. ingest-time cue-pattern path in `pam/ingestion/pipeline.py:398-468`.
**Issue:** Both paths write the same edge type but apply different mutations to the older node (status flip, importance dampening). A retrieval reading the edge cannot tell which semantics applied. Inconsistent behavior on what should be a single semantic operation.
**Fix sketch:** centralize the supersession side effects in one helper and have both paths call it; or treat ingest-cue edges as `status="proposed"` and require a follow-up confirm.

### O4 — `format_for_context_window` truncation drops endpoint nodes referenced by kept relationships *(closed 2026-05-07)*
**Where:** `pam/agent_interface.py:130-140`.
**Issue:** Truncation iterates lines in order and stops at `MAX_CONTEXT_CHARS = 4000`. A relationship line can survive while the line introducing one of its endpoints gets cut, producing dangling references in the context kayo sees.
**Fix sketch:** truncate at the section level (drop whole sections least-recently-referenced) rather than line-wise, or compute referenced node IDs first and ensure their lines are kept before truncating elsewhere.
**Resolution:** Refactored the formatter to plan lines as `_LineItem(text, kind, node_id?, refs?)` records, then replaced the truncation walk with a two-pass selector: pass 1 admits structural and node lines (must-keep nodes referenced by any candidate ref are reserved first; the rest fill remaining budget); pass 2 admits ref lines only when every endpoint is present; orphan section headers are then dropped. Tested in `tests/test_cli.py::test_format_for_context_window_truncation_keeps_referenced_endpoints` and `tests/test_cli.py::test_format_for_context_window_drops_relationships_whose_endpoints_are_evicted`.

### O5 — No `LLM_PROVIDER="off"` deterministic-only mode
**Where:** `config.py:43`, `pam/ingestion/llm.py:83-110`, `pam/retrieval/query_parser.py:505-532`.
**Issue:** The architecture treats deterministic fallback as a contract, but the only way to hit it is to make the LLM call fail (missing SDK or transport error). There is no explicit knob to force-skip the LLM call. Result: no clean way to run reproducible offline evaluations and no way for kayo to opt into a "no external calls" mode.
**Fix sketch:** treat `LLM_PROVIDER` value `"off"` (or `None`) as: skip LLM call, return empty/heuristic result without raising. Add 1–2 integration tests that ingest + query in this mode and assert sensible non-empty results.

### O6 — Duplicated LLM-response extraction *(closed 2026-05-07)*
**Where:** `pam/ingestion/llm.py:47-67` and `pam/retrieval/query_parser.py:478-498` define the same `_extract_anthropic_text()` / `_extract_openai_text()` helpers.
**Issue:** Maintenance hazard. If the upstream Anthropic or OpenAI SDK changes its response shape, two call sites diverge.
**Resolution:** moved to `pam/llm_clients.py` as `extract_anthropic_text` and `extract_openai_text`, exported via `__all__`. Both call sites now import from the shared module. 6 unit tests in `tests/test_llm_clients.py` (`ExtractAnthropicTextTests`, `ExtractOpenaiTextTests`) lock the shape contract.

### O7 — `MAX_GRAPH_DEPTH = 2` is fixed; no shape-aware adaptation *(reframed and partially closed 2026-05-07)*
**Where:** `config.py:11`, `pam/retrieval/search.py` (calls `expand()` with the constant).
**Issue:** Influence / evolution / theme questions need depth ≥ 3; lookups need depth 1. A constant can't serve both. The intended graph-native architecture explicitly calls these out as separate query shapes.
**Fix sketch:** read `parsed.intent` / `parsed.answer_mode` and pick a depth; warn (in `query_meta`) when expansion saturates.
**Reframe (2026-05-07):** the audit's framing was wrong on inspection.
1. `MAX_GRAPH_DEPTH` is **dead** — defined in `config.py` but read nowhere. Removed.
2. The expander already has limited shape-awareness via `_should_expand_related_concepts` (intent == `reason` triggers a RELATED-chain hop). It is not parameterised by a depth integer; traversal is a fixed pattern.
3. Pre-O7 IRL baseline (2026-05-07) was `34/38 = 89.5%` with `multihop_2 5/5`, `multihop_3 3/3`, `multihop_4 1/1` — all at ceiling. The categories the audit predicted depth-tuning would help are already passing. Misses cluster on answer-prompt issues (wrong_premise, demanding) and one colloquial-role retrieval gap (Q24), none of which deeper traversal would fix.
4. Broadening `_should_expand_related_concepts` to cover `influence/evolution/theme/gap` shapes was considered and **not pursued** — no failing query points at the missing expansion, and adding speculative candidates risks displacing correct answers via ranking noise. Revisit only when a concrete failure surfaces.
5. The real multi-hop traversal work is captured by the `TODO` at `pam/retrieval/graph_expander.py:258` and is roadmap-level, not audit-actionable.
**Resolution:** dead constant removed from `config.py`, `CLAUDE.md`, and `docs/MODULE_ROOT.md`. Sub-items O7b (broaden shape-aware trigger) and O7c (true multi-hop traversal) are not pursued today; documented above.

### O8 — No "why this result" payload in `RetrievalResult` *(closed 2026-05-07)*
**Where:** `pam/retrieval/ranker.py` — `RetrievalResult` exposes `query_meta` and `graph_explanations`, but per-node entries do not carry the score breakdown (FTS, recency, importance, entity-match bonus, graph-expansion contribution).
**Issue:** kayo cannot calibrate confidence in his answers. He sees an ordered list with no signal on whether the top hit is "highly relevant" vs. "barely passed FTS but boosted by recency."
**Fix sketch:** attach a per-node `score_components: dict[str, float]` populated during ranking; preserve in JSON serialization.
**Resolution:** `score()` now returns `(total, components)` with post-weight `{text_relevance, recency, importance, entity_bonus}`; `RetrievalResult` carries a `score_components` dict keyed by node id for every node in `ordered_nodes`. Components sum exactly to the rank-key (no ε). `graph-expansion contribution` is not modelled as a separate magnitude — graph-only nodes simply have `text_relevance == 0.0`. Tested in `tests/test_retrieval.py::test_ranker_exposes_score_components_per_surfaced_node`.
